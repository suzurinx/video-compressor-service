# Video Compressor Service

カスタム TCP プロトコルを使用した、バックグラウンドでの動画（MP4）アップロードシステムから進化し、独自開発の **MMP（Media Messaging Protocol）** に基づく双方向メディア変換・自動返却までを完全網羅した堅牢なネットワークシステムです。

データ構造を意識し、1バイト単位でのデータパッキング、固定長および可変長データの厳密なパース、外部プロセス（FFmpeg）の安全な非同期処理を徹底した設計になっています。

## 概要とネットワーク仕様

### 【Stage 1 仕様: 単方向アップロードプロトコル】
- **固定長通信ヘッダー（32バイト）**: 最初の4バイトで最大 4GB（2の32乗 バイト）までのファイルサイズを通知し、残りの28バイトはパディング（埋め草）で固定化しています。
- **安定したTCPセグメント送信**: ネットワークの最大セグメントサイズ（MSS）を考慮し、動画本体を上限 1400 バイトのパケットサイズに分割して送信します。
- **16バイト・レスポンスプロトコル**: アップロード完了後、サーバーから成否を示すステータス情報（4バイト ＋ パディング12バイト）を返送し、安全な終了を保証します。

### 【Stage 2 仕様: 可変長MMPによる双方向返却プロトコル】
- **可変長対応MMPヘッダー（8バイト固定）**: 指示書（JSON）、拡張子（メディアタイプ）、動画（ペイロード）という異なる3つの性質のデータを1パケットで送受信するため、最初の2バイトでJSON長、次の1バイトでメディアタイプ長、残りの5バイトでペイロード長を通知する独自の **MMP（Media Messaging Protocol）** を設計。
- **エラーハンドリングプロトコル**: システムエラーや不正リクエストを検知した場合、MMP仕様に基づき、メディアサイズおよびペイロードサイズを `0` に設定し、エラーコード・原因・解決策を包含したJSONファイルを送り返す仕様を策定。
- **絶対防衛受信エンジン（recv_all）**: TCPソケット通信におけるパケット細切れ（MTU 1500バイト制限）の現実を見据え、残りの必要バイト数をリアルタイムに計算・動的制限しながらループする「引き算パッキング構造」を実装。1ミリのデータの欠けもなく確実にバイナリを回収します。
- **双方向パケットバトンリレー**: サーバーで調理（変換）が終わったデータに対し、サーバー側が「MMPヘッダーの送信者」となり、クライアント側が「recv_all による受信者」となることで、非対称になりがちなダウンロード通信を安全かつ対称的なプロトコルとして確立。

## 構成
```
video-compressor-service/
├── README.md
├── server/
│   ├── stage1_server.py   # Stage 1: 単方向アップロード用サーバー
│   └── stage2_server.py   # Stage 2 & 3: 常時待機、MMP解読、FFmpeg調理、データ送り返し
└── client/
    ├── stage1_client.py   # Stage 1: 単方向アップロード用クライアント
    ├── stage2_client.py   # Stage 2 & 3: MMPリクエスト送信、処理済みデータの回収・保存
    └── sample.mp4         # 送信用テスト動画
```

## 使い方

### 1. サーバーの起動
先にサーバー側を立ち上げて、バックグラウンドで接続を待機させます。
```
[Stage 1 の場合]
cd server
python3 stage1_server.py
```
```
[Stage 2 の場合]
python server/stage2_server.py

※ 起動すると 127.0.0.1:8080 (localhost:8080) でクライアントからの接続を待ち受けます。
```
### 2. クライアントからの送信
別のターミナルから、プログラムを実行します。
```
[Stage 1 の場合]
cd client
python3 stage1_client.py

[Stage 2 の場合]
python client/stage2_client.py
```
### 3. 実行結果と提供されるマルチアクション
- **Stage 1**: クライアント側は指定されたファイルが .mp4 形式か自動チェックし、32バイトのヘッダーと動画本体（1400バイト分割）を送信後、サーバーからの16バイトのお返事を受け取って終了します。サーバー側は届いた32バイトからファイルサイズを正確に解剖し、バッファをコントロールしながら received_video.mp4 として保存します。
- **Stage 2 (FFmpegによるメディア調理)**: クライアントが接続した瞬間、2つの画面間で高速なパケットのパス回しが発生し、以下の仕様に完全適合した変換処理が行われます。
  - **動画の圧縮 (`compress`)**: サーバーの裏側で FFmpeg が `-vcodec libx264 -crf 28` を動的に制御。大きなファイルサイズの大幅な削減を実現しながらも、オリジナルに近い画質を維持した動画を自動返却。
  - **解像度の変更 (`resize`)**: 指示書に書かれた `width` と `height` へ動的にスケール変更（`scale=width:height`）を行い返却。
  - **音声への変換 (`audio`)**: 動画から映像を剥ぎ取り（`-vn`）、`-acodec libmp3lame` によって純粋な `.mp3` 音声ファイルへとコンバートして返却。
  - **アスペクト比の変更 / GIF・WEBM時間切り出し (`aspect` / `gif`)**: 共通関数 `safe_execute_ffmpeg` へのパス回しにより、パラメータに応じた拡張コマンドを安全に実行可能。
  - **自動クリーンアップ**: 処理完了後、サーバー側のストレージを汚さないよう、入力・出力の一時ファイルは全自動で安全に削除（消滅）されます。

## 非機能要件・安全設計への適合
- **完全性の保証**: TCPプロトコルと精密なバッファ制限（min(4096, left_bytes)）により、データが破損することなく、完全な状態で届くか、失敗して保存されないかの状態を徹底しています。
- **大容量対応**: 最大 4TB のデータ保存まで耐えられる、長期的な運用のためのロジックになっています。
- **高スループット**: 各ファイルシステムサーバにおいて、毎秒 20,000 個の 1400 バイトのパケットを処理する能力を前提とした、無駄のないデータ構造を選択しています。
- **窒息（デッドロック）の防止**: 通信途中でクライアントや回線が急に途絶した場合に、受信関数が手ぶら（None）で帰ってくる例外仕様を構築。その直後に `if not ... or video_payload is None` という「防衛検問所」を構えることで、存在しないデータをデコード・処理しようとするクラッシュやサーバープロセスの窒息フリーズを完璧に防衛。
- **メモリ空間のシンクロ**: データの有無（通信切れの `None` や、サイズ0バイトの正常データである `b""`）が持つ意味を厳密に書き分けて仕分けることで、実務のビジネスルールと通信の現実を100%一致させています。
