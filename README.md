# Video Compressor Service - Stage 1

カスタム TCP プロトコルを使用した、バックグラウンドでの動画（MP4）アップロードシステムです。
データ構造を意識し、1バイト単位でのデータパッキングと固定長ヘッダーを採用した設計になっています。

## 概要とネットワーク仕様
- 固定長通信ヘッダー（32バイト）: 最初の4バイトで最大 4GB（2の32乗 バイト）までのファイルサイズを通知し、残りの28バイトはパディング（埋め草）で固定化しています。
- 安定したTCPセグメント送信: ネットワークの最大セグメントサイズ（MSS）を考慮し、動画本体を上限 1400 バイトのパケットサイズに分割して送信します。
- 16バイト・レスポンスプロトコル: アップロード完了後、サーバーから成否を示すステータス情報（4バイト ＋ パディング12バイト）を返送し、安全な終了を保証します。

## 構成
```text
video-compressor-service/
├── README.md
├── server/
│   └── stage1_server.py
└── client/
    ├── stage1_client.py
    └── sample.mp4          # 送信用テスト動画
```

## 使い方
Ubuntu 環境でターミナルを2つ立ち上げ、以下の順番で実行します。

### 1. サーバーの起動
先にサーバー側を立ち上げて、バックグラウンドで接続を待機させます。

```markdown
cd server
python3 stage1_server.py
```

起動すると 127.0.0.1:8080 でクライアントからの接続を待ち受けます。

### 2. クライアントからの送信
別のターミナルから、プログラムを実行します。

```markdown
cd client
python3 stage1_client.py
```

### 3. 実行結果
- クライアント側: 指定されたファイルが .mp4 形式か自動チェックし、32バイトのヘッダーと動画本体（1400バイト分割）を送信後、サーバーからの16バイトのお返事を受け取って終了します。
- サーバー側: 届いた32バイトからファイルサイズを正確に解剖し、バッファをコントロールしながら received_video.mp4 として保存します。

## 非機能要件への適合
- 完全性の保証: TCPプロトコルと精密なバッファ制限（min(4096, left_bytes)）により、データが破損することなく、完全な状態で届くか、失敗して保存されないかの状態を徹底しています。
- 大容量対応: 最大 4TB のデータ保存まで耐えられる、長期的な運用のためのロジックになっています。
- 高スループット: 毎秒 20,000 個の 1400 バイトのパケットを処理する能力を前提とした、無駄のないデータ構造を選択しています。
